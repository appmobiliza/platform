# GitHub Workflow Helper

## Overview

This skill assists with the entire Git and GitHub workflow. It manages branch creation with standard naming conventions, analyzes repository changes to write and execute commits using the Conventional Commits pattern, and drafts detailed Pull Requests (PRs). It is designed to logically group changes into atomic commits and streamline repository management.

## Instructions

### Step 1: Understand User Intent

Comprehensively analyze the user's request and repository state to determine the current goal:
- Are we creating a new branch for a task?
- Are we committing unstaged/staged changes?
- Are we opening a Pull Request?
- Read `git diff`, `git status`, and file contents to understand the context.

**If the intent is ambiguous, use the AskUserQuestion tool:**
Situations that require clarification:
- The changes can be interpreted in multiple ways (e.g., Is this a bug fix or a feature?).
- The reason for the change is unclear.
- Branch naming lacks context (e.g., missing issue number).

Example question:
'''json

  "questions": [{
    "question": "What is the primary goal of these changes?",
    "header": "Change Intent",
    "multiSelect": false,

      { "label": "Feature", "description": "Adding new functionality" },

      { "label": "Refactor", "description": "Improving code without changing behavior" },
      { "label": "Chore/Docs", "description": "Updating documentation, config, etc." }

  }]

'''

### Step 2: Branch Management

When creating a branch, use a standard naming convention:
- Format: `<type>/<issue-number>-<short-description>` or `<type>/<short-description>`
- **Types**: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`
- **Example**: `feat/user-authentication` or `fix/123-login-crash`

### Step 3: Categorize Changes & Determine Commit Type

If there are active changes, group them logically:
1. **By Type** (based on Conventional Commits).
2. **By Feature/Module** (group related changes).
3. **By File Type** (docs, source code, config).

Assign the appropriate Conventional Commit type to each group:
- **feat**: A new feature
- **fix**: A bug fix
- **docs**: Documentation only changes
- **style**: Changes that do not affect the meaning of the code (white-space, formatting, etc.)
- **refactor**: A code change that neither fixes a bug nor adds a feature
- **test**: Adding missing tests or correcting existing tests
- **chore**: Changes to the build process or auxiliary tools and libraries
- **perf**: A code change that improves performance

### Step 4: Write Commit Messages

For each group, write a commit message in PT-BR using the following format:
'''text
<type>: <subject>

<body>
'''

**Subject Rules**:
- Keep it under 50 characters.
- Use the imperative present tense (e.g., "add" instead of "added" or "adds").
- Do not capitalize the first letter.
- No period (.) at the end.

**Body Rules** (Optional, but recommended for complex changes):
- Leave one blank line between the subject and the body.
- Wrap the body at 72 characters.
- Explain the *what* and *why* of the change, not the *how*.
- Use bullet points if there are multiple details.

### Step 5: Pull Request Management

When tasked with creating a Pull Request, generate a clear and descriptive title and body.

**PR Title**: Should summarize the entire branch (often matches the main commit's subject).
**PR Body Format**:
'''markdown
## Description
Provide a detailed description of the changes. Explain the problem being solved and how this PR addresses it.

## Changes Made
- Point 1
- Point 2

## Related Issues
Closes #<issue-number> (if applicable)
'''

### Step 6: Execution Guidelines

Execute the workflow sequentially based on the task:

1. **Branching (Bash)**:
   '''bash
   git checkout -b <branch-name>
   '''

2. **Staging & Committing (Bash)**:
   '''bash
   git add <file1> <file2>
   git commit -m "$(cat <<'EOF'
   <type>: <subject>

   <body>
   EOF
   )"
   '''

3. **Pushing & PR Creation (Bash via GitHub CLI)**:
   '''bash
   git push -u origin HEAD
   gh pr create --title "<PR Title>" --body "<PR Body>"
   '''

### Step 7: Final Report

After completing the requested actions:
1. Summarize the branch created, commits made, or PR opened.
2. Confirm success using `git status`, `git log -1`, or `gh pr view`.
3. Ask the user if any further actions are required.

## Crucial Constraints
- Write ALL outputs, branch names, commit messages, and PR descriptions strictly in **PT-BR**.
- NEVER mention AI tools (e.g., "Generated by Claude", "AI drafted").
- Do NOT include `Co-Authored-By` lines unless explicitly requested.
- Keep commits atomic (one logical change per commit).
- Sound natural, like a professional developer writing their own logs.
